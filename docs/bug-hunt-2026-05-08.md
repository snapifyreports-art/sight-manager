## Fixed and shipped

### `dae5049` — Snag + Order endpoints enforce site-access on every read/write
- **P1 security**: `GET /api/snags/[id]`, `PATCH /api/snags/[id]`, `DELETE /api/snags/[id]` were returning/mutating without checking the caller could see the plot's site. Any authenticated user could read or modify any snag whose id they guessed.
- **P1 security**: `GET /api/orders/[id]` was unconditionally returning the order. `PUT` only checked site access on cross-site `jobId` reassignment — same-job edits (status flip, supplier change, dates) bypassed the check.
- Fix: every handler now resolves the owning siteId and runs `canAccessSite`. Returns 404 (not 403) so unprivileged callers can't probe for record existence.

### `0f89b01` — Timezone drift + weekend-snap + silent CreateOrder failure
- **P1**: `JobWeekPanel.tsx:970` used `addWorkingDays(...).toISOString().slice(0,10)` to format a local-midnight result as a date key — UK-BST shifts it back a day. Same class of bug as the Pull Forward fix from April. Replaced with `toDateKey()`.
- **P1**: Auto-reorder (`actions/route.ts`), bulk-status (`bulk-status/route.ts`), and Pull Forward preflight (`pull-forward/route.ts`) all add lead time as calendar days (correct — supplier lead time is wall-clock, not working days) but didn't snap the result to a working day. So `expectedDeliveryDate` could land on a Saturday. apply-template-helpers already snapped; the three sister paths didn't. Added `snapToWorkingDay(..., "forward")`.
- **P1 UX**: `CreateOrderDialog.handleCreate` had an empty `else` branch on `res.ok=false` — silent failure, no error toast, button just re-enabled. User would plausibly retry and create duplicates. Now surfaces the error and leaves form state intact.

### `d2fccd4` — NaN guards on parseFloat/parseInt
- **P2**: `parseFloat(quantity)` / `parseFloat(unitCost)` / `parseInt(leadTimeDays)` returned NaN for malformed bodies (`"1.2.3"`, `"abc"`). NaN propagated into the DB; every downstream `qty * unitCost = totalCost` calc became NaN, silently breaking analytics and budget reports. Added `Number.isFinite(n) && n >= 0` guards across all four order-mutation routes; bad input now falls back to defaults (or to existing stored value on PUT).
- No live data corruption observed (the editor's controlled number inputs wouldn't allow this), but a direct API client could trigger it.

## Investigated but NOT a bug

The agents flagged several issues that turned out to be misreads:

- **"Sign-off doesn't recompute parent"** — `recomputeParentOf(prisma, id)` IS called for *every* action (line 306 of `actions/route.ts`), including signoff. Agent counted only the explicit branches.
- **"buildCompletePercent missing on signoff"** — sign-off never changes a Job's status (job is already COMPLETED before signoff). The percentage is unchanged.
- **"cascade.ts dateOfOrder null-safety"** — the schema declares `dateOfOrder DateTime @default(now())` (non-null with default). `CascadeOrder` type matches; no nulls possible at runtime.
- **"Delay endpoint ON_HOLD scope mismatch"** — including ON_HOLD children in parent date rollup is intentional. ON_HOLD means "paused, not removed"; the parent stage still spans them. Different semantics than COMPLETED.
- **"OrdersClient handleSave lockup on error"** — already in `try / finally { setSaving(false) }`. State always resets.
- **"Daily-brief / calendar / day-sheet date param parsing"** — the apparent UTC drift is fine in practice. Stored UTC-midnight `Date` values fall within the local-tz `startOfDay`/`endOfDay` window in BST/GMT, so the query catches them.
- **"Plot-templates POST has no org scoping"** — confirmed but **deferred**. Multi-org isn't on the roadmap (single tenant for now); revisit when a second org is needed.
